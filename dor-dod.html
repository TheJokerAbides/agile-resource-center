Definition of Done & Ready Examples

A practical reference for teams defining their agreements on when work is ready to start and when it's truly complete — at the Story, Epic, and Release level.

Overview Story DoR / DoD Epic DoR / DoD Release DoR / DoD INVEST

Definition of Ready (DoR)

An agreement of the project team that defines certain pre-conditions that must be met to know when something is ready to take action on. The conditions of the DoR should include: Target Audience, Goal(s), Required Tasks, Technical Requirements, Time Estimation, Clear Acceptance Criteria, and complement the DoD.

PurposeThe DoR provides criteria that allows the team to evaluate work in the backlog as being ready prior to initiating work on it. A well-conceived DoR will enhance communication, improve efficiency, reduce errors, promote collaboration, and empower the team.

Definition of Done (DoD)

An agreement made by the entire project team that defines the required steps to finish and deliver an incremental release with the best quality possible, from development through deployment.

PurposeAn explicit and concrete DoD is a critical checkpoint in the agile development process. The DoD should start with organisational standards that all teams must follow as a minimum. It ensures that functionality produced at the end of a sprint is of high quality with minimal defects.

Story Level
Story
Criteria applied to individual user stories before and after development
Definition of Ready
  • RIssue has a description
  • RItem has acceptance criteria
  • RItem has been prioritised and approved by the Product Owner
  • RDependencies have been identified to the best of the team's knowledge
  • RItem has been refined by the development team
  • RPoint of contact is identified as needed
  • RThe story fits within a sprint
  • RFollows INVEST criteria
  • RIssue has a Fix Version
Definition of Done
  • DCoding is completed
  • DUnit tests are done
  • DAdditional tests completed as applicable (functional, automated, performance)
  • DTested and validated in the QA environment
  • D"To Do's" are resolved
  • DDocumentation updated in Confluence
  • DCode was paired, swarmed or at minimum peer reviewed
  • DCode is checked into a branch (flow dependent)
  • DFuture state: Functional stories reviewed and accepted by Product Owner
⚡
Epic Level
Epic
Criteria applied to epics spanning multiple sprints or releases
Definition of Ready
  • REpic has a summary and details
  • RValue statement is clearly defined
  • REpic has acceptance criteria
  • RDefinition of success is clear
  • RTest scenarios have been evaluated
  • RDependencies and constraints have been identified
  • RRisks have been identified
  • REpic has been T-shirt sized (max size considered for release cadence)
  • REpic has been prioritised
  • RNFRs not part of Story DoD have been identified
  • RImpact personas are identified
  • RSMEs have been identified
  • RUnknowns have been identified and minimised
  • RRequired designs are complete
  • RArchitecture considerations are documented
Definition of Done
  • DAll stories are "Done"
  • DFeature has been accepted by the Product Owner
  • DRegression testing is complete in production-like environment
  • DStatic and dynamic scans are complete
  • DPerformance testing is complete
  • DDDL and DML scripts are checked into a release branch
  • DDeployment package has been peer reviewed
  • DUnit tests all pass
  • DIntegration tests pass
  • DDocumentation and run books have been updated
  • DTraining completed as necessary (iMIS)
  • DUX / 508 compliance review has been completed
🚀
Release Level
Release
Criteria that must be met before and after a production release
Definition of Ready
  • RAll stories meet the Definition of Done
  • REpic has been accepted by the Product Owner
  • RRegression testing complete in STAGE environment
  • RStatic and dynamic scans are complete
  • RPerformance testing complete (as applicable)
  • RDDL and DML scripts checked into release branch
  • RDeployment package has been peer reviewed
  • RUnit tests all pass
  • RIntegration tests pass
  • RDeployment steps have been updated
  • RUser documentation and run books updated
  • RTraining completed as necessary
  • RUX / 508 compliance review completed
Definition of Done
  • DPartners have been notified
  • DRelease has been smoke tested
  • DNo severe bugs identified
  • DBusiness Owners provide acceptance

INVEST Criteria
The six qualities that define a well-formed user story — referenced in the Story DoR

Every user story should satisfy the INVEST criteria before it is considered ready. These qualities ensure stories are well-scoped, independently deliverable, and testable within a sprint.

I
Independent
Of all other stories — can be developed and delivered on its own
N
Negotiable
Not a fixed contract — details can be discussed and refined
V
Valuable
Delivers value to the end user or customer (also: vertical)
E
Estimable
The team can estimate its size to a good approximation
S
Small
Small enough to fit comfortably within a single iteration
T
Testable
In principle — even if a test doesn't exist yet
